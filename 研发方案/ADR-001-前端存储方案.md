# ADR-001: 前端存储方案选择

## 状态
Proposed

## 背景

MiniAI记事本（KINETIC_NOTES）定位为"轻量级个人信息管理中心"，核心功能包括笔记管理、分类标签、收支记录等模块。根据PRD定义的技术目标和非功能需求，存储方案需要满足以下约束：

**数据类型与规模预估：**
| 数据类型 | 字段复杂度 | 预估量级 |
|---------|-----------|---------|
| 笔记(Note) | 13个字段，含长文本 | 数百至数千条 |
| 分类(Category) | 4个字段 | 4-20个 |
| 标签(Tag) | 4个字段 | 数十个 |
| 收入记录 | 6个字段 | 数百条 |
| 支出记录 | 6个字段 | 数千条 |

**核心约束：**
1. **性能要求**：页面加载 <= 1秒，接口响应 <= 200毫秒
2. **离线可用**：无网络时核心功能仍可使用
3. **云迁移预留**：P2需求"数据云同步"，需提前设计数据层以支持后期迁移
4. **隐私优先**：用户数据本地存储，不上传敏感信息

**技术挑战：**
- 需要支持复杂查询（按分类筛选、全文搜索、时间范围查询）
- 需要支持事务操作（创建笔记同时更新分类计数）
- 需要设计可扩展的数据层架构，降低云迁移成本

## 决策

**选择 IndexedDB 作为前端存储方案，配合 Repository 抽象层设计。**

**决策理由：**

1. **容量与性能平衡**：IndexedDB 支持大容量结构化数据存储（浏览器通常分配数百MB至GB级别），异步API不阻塞主线程，满足性能要求。

2. **查询能力充足**：支持索引创建，可高效执行分类筛选、时间范围查询等操作，满足笔记搜索和收支统计需求。

3. **标准化与兼容性**：浏览器原生API，无需引入重型依赖，主流浏览器均支持，包体积影响最小。

4. **云迁移友好**：通过 Repository 抽象层隔离存储实现，后期切换为云API只需修改 Repository 实现，业务代码无感知。

**架构设计：**

```

│              (ViewModels / State Management)             │
└─────────────────────────┬───────────────────────────────┘

│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │ NoteRepo    │  │ TagRepo     │  │ FinanceRepo │      │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘      │
│         │                │                │              │
│         └────────────────┼────────────────┘              │
│                          │                               │
│              ┌───────────▼───────────┐                   │
│              │   Storage Interface   │                   │
│              │  (IStorageProvider)   │                   │
│              └───────────┬───────────┘                   │

                           │
        ┌──────────────────┼──────────────────┐
```

## 备选方案

### 方案1: IndexedDB（推荐）

**优点：**
- 浏览器原生API，零依赖，包体积影响最小
- 容量大（数百MB至GB级别），满足长期数据存储需求
- 异步API设计，不阻塞UI主线程，性能优良
- 支持索引和事务，满足复杂查询和原子操作需求
- 支持二进制数据存储（未来可扩展图片附件）
- 广泛的浏览器兼容性（Chrome、Firefox、Safari、Edge）

**缺点：**
- 原生API复杂，学习曲线陡峭，需要封装使用
- 不同浏览器实现细节存在差异，需要兼容处理
- 无内置查询语言，复杂查询需要手动实现
- 调试工具相对有限
- 版本升级迁移需要手动处理

**云迁移可行性：**
- 高：数据结构清晰，可导出为JSON格式
- Repository抽象层屏蔽底层实现，切换成本低
- 可设计增量同步机制：本地IndexedDB + 云端API双写

**风险缓解：**
- 封装统一的Database抽象类，简化API使用
- 使用Dexie.js等轻量封装库（可选，约10KB gzip）
- 设计数据库版本迁移策略

---

### 方案2: localStorage

**优点：**
- API极其简单，上手成本为零
- 100%浏览器兼容，无兼容性问题
- 同步操作，代码简洁直观
- 无需学习复杂概念

**缺点：**
- 容量严重受限（通常5MB，部分浏览器更小）
- 只能存储字符串，需要JSON序列化/反序列化
- 同步操作会阻塞主线程，大数据量时性能差
- 无索引机制，查询需要全量遍历
- 无事务支持，数据一致性难以保证
- 无法存储二进制数据（图片附件）

**云迁移可行性：**
- 中：数据量小，导出简单
- 但容量限制无法支撑产品长期发展
- 需要在迁移前重构存储层

**风险评估：**
- 容量限制是致命缺陷，预估数据量将很快触及5MB上限
- 不建议作为主存储方案，可作为配置/缓存辅助存储

---

### 方案3: SQLite-wasm

**优点：**
- 完整的SQL能力，查询能力最强
- 单文件数据库，与后端数据模型一致
- 事务支持完善，数据一致性有保障
- 支持复杂关联查询和聚合操作
- 迁移到云端数据库成本最低（数据模型一致）
- 成熟的关系型数据库，稳定可靠

**缺点：**
- WASM文件加载增加包体积（约400KB-1MB）
- 初始加载时间较长，影响首屏性能
- 需要学习SQL语法，对前端开发者有一定门槛
- 社区和生态相对较小，问题解决资源有限
- 浏览器兼容性需要考虑SharedArrayBuffer限制

**云迁移可行性：**
- 最高：数据模型与后端数据库一致
- 可直接同步SQL语句或数据库文件
- 迁移成本最低

**风险评估：**
- 包体积增加与PRD"页面加载<=1秒"目标存在冲突
- 需要评估首屏加载影响是否可接受

---

### 方案对比矩阵

| 维度 | IndexedDB | localStorage | SQLite-wasm |
|------|-----------|--------------|-------------|
| 容量 | 高 (GB级) | 低 (5MB) | 高 (GB级) |
| 性能 | 高 (异步) | 中 (同步阻塞) | 高 |
| 查询能力 | 中 (索引) | 低 (遍历) | 高 (SQL) |
| 包体积影响 | 无 | 无 | 中 (400KB+) |
| 学习成本 | 中 | 低 | 高 |
| 事务支持 | 有 | 无 | 有 |
| 云迁移成本 | 中 | 中 | 低 |
| 浏览器兼容 | 高 | 高 | 中 |
| **综合评分** | **推荐** | 不推荐 | 备选 |

## 影响

### 正面影响

1. **满足性能要求**：异步API设计确保UI响应流畅，满足"页面加载<=1秒，接口响应<=200毫秒"目标。

2. **容量可扩展**：支持大量笔记和收支记录存储，不会触及容量瓶颈。

3. **开发效率提升**：通过Repository抽象层，业务代码与存储实现解耦，便于测试和维护。

4. **云迁移路径清晰**：架构设计预留了Provider切换机制，后期云迁移只需实现CloudProvider，业务代码零改动。

5. **离线体验保障**：IndexedDB原生支持离线存储，配合Service Worker可实现完整离线功能。

6. **包体积优化**：无需引入额外依赖，保持轻量级产品定位。

### 负面影响

1. **开发复杂度增加**：需要设计和实现Repository抽象层，初期开发成本较高。

2. **调试难度**：IndexedDB调试工具相对有限，问题排查需要更多经验。

3. **数据迁移工作**：未来数据库Schema变更需要设计迁移策略。

### 风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 浏览器IndexedDB实现差异 | 中 | 中 | 使用Dexie.js封装，统一API行为 |
| 数据库版本升级失败 | 低 | 高 | 设计增量迁移脚本，保留回滚能力 |
| 首次加载性能问题 | 低 | 中 | 懒加载非核心数据，优化索引设计 |
| 云同步冲突处理 | 中 | 高 | 设计CRDT或版本向量冲突解决机制 |

## 云迁移设计

### 数据层抽象架构

为支持后期云迁移，数据层采用以下抽象设计：

**1. 存储提供者接口（IStorageProvider）**

```typescript
interface IStorageProvider {
  // 连接/初始化
  connect(): Promise<void>;
  disconnect(): Promise<void>;

  // 通用CRUD
  get<T>(collection: string, id: string): Promise<T | null>;
  getAll<T>(collection: string): Promise<T[]>;
  query<T>(collection: string, query: QuerySpec): Promise<T[]>;
  create<T>(collection: string, data: T): Promise<T>;
  update<T>(collection: string, id: string, data: Partial<T>): Promise<T>;
  delete(collection: string, id: string): Promise<void>;

  // 事务支持
  transaction<T>(operations: Operation[]): Promise<T>;

  // 同步相关
  getChanges(since: Date): Promise<ChangeSet[]>;
  applyChanges(changes: ChangeSet[]): Promise<void>;
}
```

**2. Repository模式实现**

```typescript
// 业务层使用的Repository接口
interface INoteRepository {
  findById(id: string): Promise<Note | null>;
  findByCategory(category: string): Promise<Note[]>;
  search(keyword: string): Promise<Note[]>;
  create(note: Note): Promise<Note>;
  update(id: string, data: Partial<Note>): Promise<Note>;
  delete(id: string): Promise<void>;
}

// 实现示例
class NoteRepository implements INoteRepository {
  constructor(private storage: IStorageProvider) {}

  async findById(id: string): Promise<Note | null> {
    return this.storage.get<Note>('notes', id);
  }
  // ...其他方法实现
}
```

**3. 同步状态模型**

为支持增量同步，数据模型增加同步字段：

```typescript
interface SyncableEntity {
  id: string;
  createdAt: Date;
  updatedAt: Date;
  syncStatus: 'pending' | 'synced' | 'conflict';
  localVersion: number;
  serverVersion?: number;
}
```

### 迁移策略

**Phase 1（当前）：纯本地存储**
- IndexedDB存储所有数据
- Repository抽象层就位
- 数据模型包含同步预留字段

**Phase 2（云迁移）：本地优先 + 云同步**
- 实现CloudProvider
- 增量同步机制：本地写 -> 后台同步
- 冲突解决策略：Last-Write-Wins 或 用户选择

**Phase 3（可选）：云优先模式**
- 支持用户切换存储模式
- 本地缓存优化离线体验
- 多设备实时同步

### 迁移检查清单

- [ ] 所有数据实体继承SyncableEntity基类
- [ ] Repository层完全抽象存储实现
- [ ] 业务代码不直接调用IndexedDB API
- [ ] 数据库Schema设计支持版本字段
- [ ] 单元测试覆盖Repository接口，非具体存储
