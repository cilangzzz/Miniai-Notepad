## 2. 前端框架选择

### 2.1 候选方案对比

| 维度 | Vue 3 | React | Svelte | Angular |
|------|-------|-------|--------|---------|
| **学习曲线** | 低 | 中 | 低 | 高 |
| **包体积** | ~40KB | ~45KB | ~2KB | ~130KB |
| **性能** | 优秀 | 优秀 | 卓越 | 良好 |
| **TypeScript支持** | 原生支持 | 良好 | 良好 | 原生支持 |
| **响应式系统** | Proxy-based | setState/Hooks | 编译时 | RxJS |
| **生态成熟度** | 高 | 极高 | 中 | 高 |
| **构建速度** | 快 | 快 | 极快 | 慢 |
| **离线/PWA支持** | 良好 | 良好 | 良好 | 良好 |
| **状态管理** | Pinia（官方） | Redux/Zustand/Jotai | Svelte Stores | RxJS/NgRx |
| **组件库生态** | 丰富 | 极丰富 | 较少 | 丰富 |

### 2.2 详细分析

#### Vue 3

**优势：**
- Composition API 提供了更灵活的代码组织方式，适合复杂业务逻辑
- 响应式系统基于 Proxy，性能优异且无需手动优化
- 单文件组件(SFC)开发体验好，模板/脚本/样式一体化
- Pinia 状态管理简洁直观，TypeScript 支持完善
- Vite 官方支持，开发构建速度快
- 学习曲线平缓，团队上手快

**劣势：**
- 相比 React 生态略小
- 自定义 UI 组件库选择相对较少

**云迁移友好性：** 高。Vue 3 的 Composition API 非常适合抽象存储层，可以通过自定义 Hook 模式封装数据访问逻辑，后期迁移只需替换底层实现。

#### React

**优势：**
- 生态最丰富，组件库选择多
- 社区活跃，问题解决方案多
- React Native 可复用逻辑到移动端
- Hooks 模式成熟

**劣势：**
- 需要手动处理依赖数组，心智负担较大
- 包体积相对较大
- 状态管理方案选择过多，容易产生技术债务

**云迁移友好性：** 高。自定义 Hook 模式同样支持存储层抽象。

#### Svelte

**优势：**
- 编译时框架，运行时体积最小
- 性能最佳，适合性能敏感场景
- 语法简洁，开发体验好

**劣势：**
- 生态相对不成熟，UI 组件库少
- 大型项目实践经验较少
- 社区规模较小

**云迁移友好性：** 中。Svelte Stores 是优秀的状态管理方案，但生态工具对云迁移的支持不如 Vue/React 成熟。

#### Angular

**优势：**
- 企业级框架，架构完整
- TypeScript 原生支持
- 适合大型团队协作

**劣势：**
- 包体积大，影响首屏加载
- 学习曲线陡峭
- 对快速迭代项目不够灵活

**云迁移友好性：** 高。依赖注入模式天然支持存储层抽象。但框架整体过重，不适合本项目性能要求。

### 2.3 框架选型决策

| 决策项 | 推荐方案 |
|--------|----------|
| **主框架** | Vue 3 |
| **备选方案** | React |

**决策理由：**

1. **性能达标**：Vue 3 + Vite 组合可实现冷启动 < 500ms，热更新 < 100ms，满足页面加载 ≤ 1秒要求
2. **开发效率**：Composition API + SFC 模式适合快速迭代
3. **状态管理**：Pinia 官方支持，TypeScript 友好
4. **存储层抽象**：VueUse 工具库提供 useStorageAsync 等抽象，便于云迁移
5. **团队学习成本**：相对较低，适合快速启动项目

## 3. 前端存储方案

### 3.1 候选方案对比

| 维度 | IndexedDB | localStorage | SQLite-WASM |
|------|-----------|--------------|-------------|
| **存储容量** | ~50MB-无限制 | ~5MB | ~50MB-无限制 |
| **数据结构** | NoSQL（对象存储） | Key-Value（字符串） | 关系型数据库 |
| **查询能力** | 索引查询 | 无 | SQL完整查询 |
| **事务支持** | 支持 | 不支持 | 完整支持 |
| **异步API** | 原生异步 | 同步阻塞 | 异步 |
| **浏览器支持** | 现代浏览器 | 全部支持 | 现代浏览器 |
| **复杂度** | 中 | 低 | 高 |
| **离线支持** | 优秀 | 优秀 | 优秀 |
| **云迁移成本** | 中 | 低 | 中 |

### 3.2 云迁移友好性深度分析

#### IndexedDB 方案

**架构设计：**
```
应用层
    |
    v
Repository 抽象层（接口定义）
    |
    v
IndexedDB 适配器 <---> 同步引擎 <---> 云端API
```

**迁移策略：**
1. 定义统一的 Repository 接口（CRUD + Query）
2. IndexedDB 实现作为本地存储适配器
3. 云迁移时新增 API 适配器，实现双写双读
4. 同步引擎负责增量同步和冲突解决

**优势：**
- 支持复杂查询（索引、范围查询）
- 容量大，可存储大量笔记和附件
- 原生异步，不阻塞主线程
- 支持结构化数据，适合笔记、财务等实体

**劣势：**
- API 复杂，需要封装
- 事务处理较繁琐
- 调试困难

**推荐封装库：**
- **Dexie.js**：IndexedDB 的 Promise 封装，API 简洁，TypeScript 支持好
- **idb**：轻量级 Promise 封装，体积小

#### localStorage 方案

**架构设计：**
```
应用层
    |
    v
JSON 序列化/反序列化层
    |
    v
localStorage（单一Key或分片Key）
```

**迁移策略：**
1. 数据序列化为 JSON 字符串存储
2. 云迁移时直接 POST JSON 到服务端
3. 实现增量同步需要额外设计版本机制

**优势：**
- API 简单，学习成本低
- 兼容性最好
- 数据迁移最简单（直接导出 JSON）

**劣势：**
- 容量限制严重（~5MB），无法存储大量数据
- 同步阻塞主线程，影响性能
- 无查询能力，需要全量加载到内存
- 无法存储二进制数据（图片附件）

**结论：** 不适合作为主存储方案，可作为配置/缓存辅助存储。

#### SQLite-WASM 方案

**架构设计：**
```
应用层
    |
    v
ORM 抽象层
    |
    v
SQLite-WASM（浏览器内SQLite）
```

**迁移策略：**
1. SQLite 数据库文件可导出
2. 服务端可直接使用相同 Schema
3. 增量同步可通过时间戳字段实现

**优势：**
- 完整的 SQL 查询能力
- 事务支持完善
- 与后端数据库 Schema 一致
- 适合复杂查询场景

**劣势：**
- 初始化体积较大（~1MB）
- 学习成本高
- 浏览器兼容性需要考虑
- 过度设计风险（本项目查询需求简单）

### 3.3 混合存储策略推荐

考虑到本项目的实际需求，推荐采用**混合存储策略**：

| 数据类型 | 存储方案 | 理由 |
|----------|----------|------|
| 笔记内容、财务记录 | IndexedDB (Dexie.js) | 需要结构化存储和查询 |
| 用户配置、UI状态 | localStorage | 容量小，访问频繁 |
| 新闻缓存 | IndexedDB + Cache API | 需要离线访问 |
| 会话数据 | sessionStorage | 临时数据 |

### 3.4 存储架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                       应用业务层                              │
├─────────────────────────────────────────────────────────────┤
│                   Repository 接口层                           │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │ NoteRepo     │  │ FinanceRepo  │  │ ConfigRepo   │       │
│  │ (interface)  │  │ (interface)  │  │ (interface)  │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
├─────────────────────────────────────────────────────────────┤
│                   存储适配器层                                │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │ IndexedDB    │  │ localStorage │  │ Memory       │       │
│  │ Adapter      │  │ Adapter      │  │ Adapter      │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
├─────────────────────────────────────────────────────────────┤
│                   同步引擎（预留）                            │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ SyncEngine: 增量同步 | 冲突检测 | 离线队列            │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 3.5 存储方案决策

| 决策项 | 推荐方案 |
|--------|----------|
| **主存储** | IndexedDB (Dexie.js 封装) |
| **辅助存储** | localStorage (配置/状态) |
| **缓存策略** | Cache API (静态资源) + IndexedDB (数据) |

**决策理由：**

1. **容量充足**：IndexedDB 容量远大于 localStorage，支持用户长期使用
2. **查询能力**：支持索引查询，满足笔记搜索、财务统计需求
3. **异步非阻塞**：保证 UI 响应性能
4. **云迁移友好**：Repository 模式便于后期替换存储实现
5. **Dexie.js 封装**：降低 IndexedDB API 复杂度，提供友好的 Promise API

## 4. 状态管理方案

### 4.1 候选方案对比

| 维度 | Pinia | Vuex 4 | Zustand | Jotai |
|------|-------|--------|---------|-------|
| **所属生态** | Vue官方 | Vue官方 | React | React |
| **TypeScript** | 优秀 | 一般 | 优秀 | 优秀 |
| **学习曲线** | 低 | 中 | 低 | 中 |
| **DevTools** | 支持 | 支持 | 支持 | 支持 |
| **代码量** | 少 | 多 | 少 | 少 |
| **模块化** | 天然支持 | 需要配置 | 灵活 | 灵活 |
| **持久化** | 插件支持 | 插件支持 | 插件支持 | 插件支持 |

### 4.2 Vue 生态状态管理分析

#### Pinia（推荐）

**优势：**
- Vue 3 官方推荐，Composition API 天然集成
- 完整的 TypeScript 支持，类型推导完善
- 无需 mutations，代码简洁
- 支持 Vue DevTools
- 模块化天然支持，无需嵌套
- 持久化插件生态成熟（pinia-plugin-persistedstate）

**代码示例：**
```typescript
// stores/note.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import type { Note } from '@/types'
import { noteRepository } from '@/repositories/note'

export const useNoteStore = defineStore('note', () => {
  const notes = ref<Note[]>([])
  const loading = ref(false)

  const pinnedNotes = computed(() =>
    notes.value.filter(n => n.is_pinned)
  )

  async function fetchNotes() {
    loading.value = true
    notes.value = await noteRepository.findAll()
    loading.value = false
  }

  return { notes, loading, pinnedNotes, fetchNotes }
}, {
  persist: {
    storage: sessionStorage, // 可选持久化
    paths: ['notes'] // 指定持久化字段
  }
})
```

#### Vuex 4

**优势：**
- Vue 2/3 兼容
- 成熟稳定

**劣势：**
- 需要 mutations，代码冗余
- TypeScript 支持不如 Pinia
- 模块化配置复杂
- 官方已推荐 Pinia 作为首选

**结论：** 对于新项目，Vuex 4 不如 Pinia 适合。

### 4.3 状态管理架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                       Pinia Stores                           │
├──────────────┬──────────────┬──────────────┬───────────────┤
│  noteStore    │  financeStore│  configStore │  newsStore    │
│  - notes      │  - incomes   │  - theme     │  - articles  │
│  - categories │  - expenses  │  - settings  │  - cache     │
│  - tags       │  - budget    │  - user      │              │
├──────────────┴──────────────┴──────────────┴───────────────┤
│                   Composables (业务逻辑封装)                  │
│  useNotes(), useFinance(), useSearch(), useSync()           │
├─────────────────────────────────────────────────────────────┤
│                   持久化插件                                   │
│  pinia-plugin-persistedstate (可选持久化)                    │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 状态管理决策

| 决策项 | 推荐方案 |
|--------|----------|
| **状态管理** | Pinia |
| **持久化** | pinia-plugin-persistedstate（可选） |
| **业务逻辑** | Composables 封装 |

**决策理由：**

1. **官方推荐**：Pinia 是 Vue 3 官方状态管理库
2. **类型安全**：TypeScript 支持完善，开发体验好
3. **简洁直观**：相比 Vuex 减少大量样板代码
4. **DevTools 支持**：调试方便
5. **持久化灵活**：可选择性地持久化特定 store

## 5. UI组件方案

### 5.1 候选方案对比

| 方案 | 定制程度 | 开发效率 | 包体积 | Neo-Brutalist适配 |
|------|----------|----------|--------|-------------------|
| 自定义组件 | 高 | 低 | 最小 | 最佳 |
| Tailwind CSS | 高 | 高 | 小 | 优秀 |
| Element Plus | 低 | 高 | 大 | 困难 |
| Ant Design Vue | 低 | 高 | 大 | 困难 |
| Naive UI | 中 | 高 | 中 | 中等 |

### 5.2 Neo-Brutalist 设计要求分析

根据 PRD 设计系统要求：

| 设计特征 | 技术要求 |
|----------|----------|
| 0px 圆角 | 需要 override 组件库默认样式 |
| 4px 粗边框 | 需要 override 组件库默认样式 |
| 8px 偏移阴影 | 自定义阴影实现 |
| 倾斜元素 | CSS transform 实现 |
| Hover 上移交互 | CSS transition 实现 |
| 深色主题 (#131313) | 深色主题支持 |

**结论：** 现有 UI 组件库的默认样式与 Neo-Brutalist 风格差异巨大，Override 成本高，不如自定义组件灵活。

### 5.3 推荐方案：Tailwind CSS + 自定义组件

### 5.5 UI方案决策

| 决策项 | 推荐方案 |
|--------|----------|
| **样式方案** | Tailwind CSS |
| **组件策略** | 自定义组件库 |
| **图标库** | Lucide Icons / Heroicons |
| **动画** | Tailwind 内置 + CSS transitions |

**决策理由：**

1. **设计独特性**：Neo-Brutalist 风格高度定制，现成组件库无法满足
2. **包体积最优**：Tailwind CSS 按需生成，自定义组件无冗余
3. **开发效率**：Tailwind 原子类快速开发，组件复用提高效率
4. **维护性**：设计系统集中在 tailwind.config.js，便于统一管理
5. **性能**：无第三方组件库运行时开销

### 7.2 数据模型设计（云同步友好）

interface BaseEntity {

### 7.3 同步策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **增量同步** | 基于 updated_at 时间戳，只同步变更数据 | 日常同步 |
| **Last-Write-Wins** | 冲突时取最新时间戳的数据 | 简单场景 |
| **软删除** | 删除操作标记 deleted_at，同步后清理 | 数据一致性 |
| **离线队列** | 离线操作存入队列，联网后自动同步 | 离线场景 |
| **全量同步** | 首次登录或数据异常时全量拉取 | 恢复场景 |

### 7.5 同步引擎接口设计

```typescript
// services/sync/engine.ts

interface SyncEngine {
  // 启动同步引擎
  start(): Promise<void>

  // 停止同步引擎
  stop(): Promise<void>

  // 手动触发同步
  sync(): Promise<SyncResult>

  // 获取同步状态
  getStatus(): SyncStatus

  // 注册冲突处理器
  onConflict(handler: ConflictHandler): void
}

interface SyncResult {
  success: boolean
  pulled: number   // 拉取数量
  pushed: number   // 推送数量
  conflicts: Conflict[]
  errors: Error[]
}

type SyncStatus = 'idle' | 'syncing' | 'synced' | 'error' | 'offline'
```

## 9. 决策总结

### 9.1 决策矩阵

| 决策项 | 选择 | 备选 | 选择理由 |
|--------|------|------|----------|
| 前端框架 | Vue 3 | React | 性能优秀、开发效率高、生态成熟 |
| 存储方案 | IndexedDB (Dexie.js) | SQLite-WASM | 容量充足、查询能力强、云迁移友好 |
| 状态管理 | Pinia | Vuex | 官方推荐、TypeScript 友好、代码简洁 |
| UI方案 | Tailwind + 自定义组件 | Element Plus | Neo-Brutalist 风格高度定制需求 |
| 构建工具 | Vite | Webpack | 开发体验最佳、Vue 官方支持 |

### 9.2 关键决策依据

1. **性能优先**：Vue 3 + Vite + Tailwind 组合确保首屏加载 ≤ 1秒
2. **离线支持**：IndexedDB + PWA 实现完整离线能力
3. **云迁移友好**：Repository 抽象层 + 同步引擎预留架构
4. **设计独特性**：自定义 Neo-Brutalist 组件库
5. **开发效率**：成熟的工具链 + 简洁的状态管理
