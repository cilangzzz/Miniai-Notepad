# 用国产大模型 + Claude Code 复现记账软件：一次 AI 驱动开发的实践

## 前言

本文记录了使用国产模型复现"盒友"记账软件的完整过程，旨在验证国内模型在实际软件开发中的可行性。

**使用工具**：
- GLM-5 + Claude Code（全程零手敲，纯 AI 生成）
- [software-dev-ai-workflow](https://github.com/cilangzzz/software-dev-ai-workflow) — 一套完整的 AI 驱动软件开发工作流 Skill 集合，覆盖从需求到运维的全流程

**项目代码**：[Miniai-Notepad](https://github.com/cilangzzz/Miniai-Notepad) — 一个极简记事本项目，用于验证 AI 工作流程

**当前状态**：核心功能已复现，部分功能仍在调试修复中（主要因为功能点设计时描述不够详细）。

> **说明**：作者此前未使用过记账类软件，业务需求了解有限；UI 设计基于盒友提供的一张截图还原。

---

## 开发流程

整个流程分为**需求设计阶段**和**分阶段实现阶段**：

### 需求设计阶段

| 步骤 | 内容 | 备注 |
|------|------|------|
| 项目概述 | 定位、核心模块、用户画像 | 由产品 Agent 设计，仅一句话描述 |
| 需求分析 | PRD 编写、用户故事、设计规范 | — |
| 技术选型 & 系统架构 | 纯前端实现，存储设计为可切换云存储 | 明确约束技术栈 |
| 功能点设计 | 各功能点详细设计 | **关键环节：约束越多，AI 理解越准确** |
| UI 设计 | 参考截图 → Google Stitch → 官方 Skill 转换为风格组件 | — |

### 分阶段实现阶段

为避免上下文溢出，每个阶段由独立的 Agent 处理：

## 后续调试

实现完成后进入 Debug 阶段。发现部分页面按钮存在问题：原型设计出来了，但功能点设计文档中缺少对应功能的详细描述，导致实现偏差。

---

## 总结

这次实践验证了国产大模型配合 Claude Code 实现中小型项目的可行性。**功能点设计阶段的质量直接影响最终实现效果**——描述越详细、约束越明确，AI 的产出就越符合预期。
